# ML Autoresearch — Complete Explanation

## What Is It

The ML Autoresearch system is an autonomous machine learning optimization loop where an LLM (Claude Sonnet) acts as a researcher, iteratively modifying a neural network surrogate model to minimize prediction error on battery cell performance data. Over 200 experiments across 18+ hours, the system achieved a 57.5% improvement in prediction accuracy — with the key breakthroughs coming from physics-informed feature engineering that a traditional hyperparameter optimizer would not discover.

---

## Purpose

FORGE models 4680-format cylindrical lithium-ion battery cells (the format used in Tesla Model Y, BMW Neue Klasse, etc.). Engineers need to predict how a cell design will perform (rate capability, thermal behavior) before building physical prototypes.

The surrogate model replaces expensive physics simulations (PyBaMM, COMSOL) with a fast neural network approximation. Given 11 geometric design parameters, it predicts:

1. **Rate capability proxy** — how well the cell handles high discharge rates (higher = better)
2. **Max temperature proxy** — peak temperature during fast charging (lower = better, in °C)

The autoresearch loop's job is to find the best possible neural network architecture and feature set for this prediction task — autonomously, without human intervention.

## The Dataset

### Generation

The dataset is synthetic, generated by `prepare.py` using physics-based formulas that capture the key relationships in battery cell design:

- **Sampling:** 2000 cell designs via Latin Hypercube Sampling (ensures uniform coverage of the design space)
- **Split:** 1400 train / 300 validation / 300 test (70/15/15)
- **Integrity:** SHA-256 hashes for all split files, verified before every training run

### Input Features (11 total)

**8 geometric parameters** (directly sampled):

| Feature | Range | Unit | Physical Meaning |
|---------|-------|------|------------------|
| electrode_thickness | 50–150 | μm | Active material layer thickness |
| porosity | 0.20–0.50 | — | Electrolyte volume fraction |
| separator_thickness | 10–50 | μm | Ion-permeable membrane thickness |
| n_tabs | 1–6 | count | Current collector connection points |
| tab_width | 5–30 | mm | Width of each current tab |
| can_inner_diameter | 44–46 | mm | Cell casing inner diameter (narrow range) |
| can_wall_thickness | 0.2–0.8 | mm | Casing wall thickness |
| cell_height | 65–95 | mm | Total cell height |

**3 derived features** (computed from geometric parameters):

| Feature | Formula | Physical Meaning |
|---------|---------|------------------|
| surface_to_volume | 2/(r) + 2/(h) | Thermal dissipation capacity |
| tab_conductance_proxy | n_tabs × tab_width | Total current collection area |
| diffusion_path_proxy | electrode_thickness / porosity | Effective ion transport distance |

### Target Physics

The target formulas encode real electrochemical relationships:

**Rate capability** involves:
- Butler-Volmer kinetics (tanh saturation of tab conductance)
- Sigmoid diffusion limitation (threshold at ~350 μm effective path length)
- Separator resistance penalty
- Electrode-porosity interaction

**Max temperature** involves:
- Inverse surface-to-volume scaling (smaller cells cool better)
- Electrode thickness threshold effect (softplus around 100 μm — thicker electrodes generate more internal heat)
- Thermal mass contribution (can wall × height)
- Tab count penalty (fewer tabs = higher local current density = more heat)

Both targets include Gaussian noise (σ=0.01 for rate, σ=0.5°C for temperature) to simulate measurement uncertainty.

## The Autoresearch Loop

### Architecture

The system has three components:

1. **The Engine** (`forge/ml/autoresearch/`) — framework code that runs experiments, parses metrics, decides accept/reject, logs results. Never modified during a run.

2. **The Surrogate** (`experiments/ml/autoresearch/surrogate.py`) — the neural network training script. This is the ONLY file the LLM modifies. Contains the model architecture, feature engineering, hyperparameters, and training loop.

3. **The Agent** (Claude Sonnet via CLI) — reads the program guide, understands the physics domain, proposes modifications to surrogate.py, evaluates results, and decides next steps. Runs with `--dangerously-skip-permissions` for autonomous operation.

### How It Works

```
┌─────────────────────────────────────────────────────────┐
│                    LLM Agent (Sonnet)                    │
│                                                          │
│  1. Read current surrogate.py and experiment history     │
│  2. Hypothesize: "Bruggeman tortuosity (por^1.5)         │
│     should capture ion transport nonlinearity"           │
│  3. Edit surrogate.py to add the feature                │
│  4. Run: python surrogate.py --dataset ... --budget 300 │
│  5. Parse output metrics after "---" marker              │
│  6. Compare: new_score < best_score - 0.0001?            │
│     YES → git commit (accept), update best               │
│     NO  → git revert (reject), try something else        │
│  7. Repeat from step 1                                   │
└─────────────────────────────────────────────────────────┘
```

### Acceptance Criteria

- **Metric:** `primary_score = rmse_rate_norm + rmse_temp_norm` (lower is better)
- **Normalization:** RMSE divided by target standard deviation, so both targets contribute equally regardless of scale
- **Minimum improvement:** 0.0001 (prevents noise-driven acceptance)
- **Tie-breaking:** If scores are equal within epsilon, faster training time wins

### Constraints on the Agent

The LLM can ONLY modify `surrogate.py`. It cannot:
- Change the dataset (files are SHA-256 verified)
- Change the evaluation metrics
- Change the output format
- Change the random seed
- Use models that exceed the time budget

This ensures every improvement is genuine — the agent can't game the metric.

## Run 001 — Results

### Timeline

- **Duration:** ~18 hours
- **Experiments attempted:** 200
- **Experiments accepted:** 36 (18% acceptance rate)
- **Experiments rejected:** 164

### Score Progression

```
Baseline (exp 2):     0.3351
After architecture:   0.2937  (exp 5, skip connections)
After features:       0.1983  (exp 53, log transforms + Bruggeman)
After target norm:    0.1609  (exp 105, target normalization)
After hyperparams:    0.1517  (exp 184, ensemble=30, hidden=80, batch=32)
Final:                0.1517  (57.5% improvement from baseline)
```

| Exp | Change | Score | Δ | Category |
|-----|--------|-------|---|----------|
| 2 | GELU activation | 0.3351 | baseline | Architecture |
| 3 | Feature interactions (et×por, ntabs×h) | 0.3107 | -0.0245 | Feature Engineering |
| 5 | Skip connections (input concatenated to output) | 0.2937 | -0.0170 | Architecture |
| 10 | hidden_size=96 | 0.2821 | -0.0116 | Hyperparameter |
| 29 | BatchNorm | 0.2409 | -0.0413 | Architecture |
| 43 | log(tab_conductance) feature | 0.2175 | -0.0234 | Feature Engineering |
| 53 | log(porosity) — Bruggeman physics | 0.1983 | -0.0192 | Feature Engineering |
| 81 | AdamW with weight_decay=1e-4 | 0.1876 | -0.0107 | Training Strategy |
| 105 | Target normalization | 0.1609 | -0.0266 | Training Strategy |
| 144 | electrode_thickness² feature | 0.1581 | -0.0028 | Feature Engineering |

| 184 | ensemble=30 (final best) | 0.1517 | -0.0038 | Hyperparameter |

### Confirmed Dead Ends

The agent tried and rejected these approaches:
- Cosine annealing LR schedule
- 3-layer networks (overfitting)
- hidden_size ≥ 88 (diminishing returns)
- ensemble ≥ 50 (too many models, too little training time each)
- Weight decay ≠ 1e-4
- log(n_tabs), log(separator) features
- Dual-head architecture
- batch_size = 16 or 64
- Gradient clipping
- Input noise augmentation
- Dropout

### Final Model Architecture

```
2-layer MLP, BatchNorm, input skip connection
HIDDEN_SIZE=80, BATCH_SIZE=32, LR=5e-3
WEIGHT_DECAY=1e-4, ACTIVATION=GELU, LOSS=MSE
ENSEMBLE_SIZE=30, Device=CUDA (RTX 3060)

Features (22 total):
  11 raw inputs (can_inner_diameter zeroed)
  + et×por                    — electrode-porosity interaction
  + ntabs×height              — tab count scales with cell height
  + log(diffusion_path)       — compresses nonlinear range
  + log(tab_conductance)      — compresses conductance range
  + log(surface_to_volume)    — thermal dissipation log scale
  + log(porosity)             — Bruggeman tortuosity base
  + log(cond) - log(diff)     — conductance-to-diffusion ratio
  + electrode_thickness²      — threshold effect
  + porosity^1.5              — Bruggeman tortuosity formula
  + et / porosity^1.5         — corrected diffusion path
  + log(electrode_thickness)  — thickness log transform
```

### Why Physics-Informed Features Mattered

The four final breakthroughs (experiments 144-156) were all physics-informed:

1. **electrode_thickness²** — captures the threshold effect where thermal generation increases superlinearly above ~100 μm
2. **porosity^1.5** — the Bruggeman relation for effective ionic conductivity in porous media, a well-known electrochemical formula
3. **et/porosity^1.5** — the corrected diffusion path length accounting for tortuosity
4. **log(electrode_thickness)** — compresses the nonlinear thickness-temperature relationship

These features encode domain knowledge that a pure hyperparameter optimizer (like Optuna) could not discover — it can tune numbers, but it cannot invent physics formulas.

## Planned: Run 002 — Optuna Comparison

### Purpose

Demonstrate that LLM-guided research outperforms traditional automated hyperparameter optimization on this task. The key claim: an LLM can discover physics-informed features, while Optuna can only tune numbers.

### Design

- **Same dataset, same budget, same acceptance criteria**
- **Optuna controls:** hidden_size, learning_rate, batch_size, weight_decay, ensemble_size, activation, num_layers
- **Optuna does NOT do:** feature engineering, architecture invention, physics-informed modifications
- **Branch:** `autoresearch/run-002-optuna`
- **Budget:** 200 experiments (same as Run 001 for fair comparison)

### Expected Outcome

Optuna will likely achieve ~0.16-0.18 (good hyperparameter tuning) but will NOT reach 0.1517 because it cannot discover the Bruggeman tortuosity features that account for the final 10-15% of improvement.

### Dashboard Integration
